# Vera Harness capability expansion: protocol spike

Status: alpha design record, 2026-07-11

This note is the short research spike required before implementation. It records
the protocol choices used by the current work and the fixture behaviors that
must remain deterministic in tests. It deliberately does not add persistent
memory or session search.

## MCP

Vera uses the MCP stdio transport in this phase. A client starts one configured
child, sends one `initialize` request, waits for the negotiated response, sends
`notifications/initialized`, and only then calls `tools/list` or `tools/call`.
The alpha client advertises the stable `2025-03-26` legacy handshake revision
and accepts the server's negotiated version before using the session.
The client closes stdin during shutdown and waits briefly before terminating a
child. Requests use JSON-RPC ids; notifications and responses that do not match
the current request are ignored or routed to their pending request.

The transport boundary is an internal async trait so Streamable HTTP can be
added later without changing the tool registry. Vera validates that tool names
are non-empty and that input schemas are object schemas before exposing them to
the model. Tool results preserve text, structured content, resource links, and
image metadata in a bounded textual representation. MCP server startup and
individual calls remain separate permission actions.

Sources: [MCP lifecycle](https://modelcontextprotocol.io/specification/2025-03-26/basic/lifecycle),
[MCP transports](https://modelcontextprotocol.io/specification/draft/basic/transports),
and [MCP architecture](https://modelcontextprotocol.io/docs/learn/architecture).

Fixture contract:

1. `initialize` returns the requested protocol version and advertises `tools`.
2. `tools/list` returns one read-only tool, one mutating tool, and one malformed
   schema; only valid allow-listed schemas are registered.
3. `tools/call` emits an unrelated notification before the matching response.
4. The fixture can delay, close stdout, return a malformed `tools/list`, or
   return a JSON-RPC error; Vera reports a bounded structured error, retains
   redacted stderr diagnostics after cleanup, and never includes child
   environment values.

## Browser/CDP

Browser support is opt-in. Vera first reads a configured CDP HTTP endpoint,
discovers targets, and only connects to a target returned by that endpoint. It
does not launch Chrome or attach to an arbitrary local browser. The native
client is limited to target discovery, navigation status, bounded DOM
snapshots, and screenshots; clicks, typing, evaluation, and live console
subscriptions remain delegated to an approved browser MCP server or a later
CDP slice.

Configured endpoints are versioned in `browser_cdp_endpoints`; an empty list
still permits a separately approved endpoint through the Browser permission,
while a non-empty list is an exact project allow-list.

The CDP fixture exposes `/json/version`, `/json/list`, and a websocket page
target. The first browser slice uses `Target.getTargets`, `Page.enable`,
`Runtime.enable`, `Page.captureScreenshot`, and the console event
`Runtime.consoleAPICalled`. Host changes are checked as a separate permission.

Sources: [CDP protocol overview](https://chromedevtools.github.io/devtools-protocol/),
[Target domain](https://chromedevtools.github.io/devtools-protocol/tot/Target/),
and [Runtime domain](https://chromedevtools.github.io/devtools-protocol/v8/Runtime/).

## Git worktrees

Write-capable agents use a detached worktree created from the parent’s captured
`HEAD`, with a generated branch and a path under the repository’s guarded Git
metadata directory (`.git/vera-worktrees`; linked-worktree layouts use the
guarded `.vera/worktrees` fallback). The path is generated by Vera and checked
against the repository root before use; user-supplied path fragments are never
passed to `git worktree` unchecked.
Review is diff-first: status, base revision, changed paths, tests, and conflict
state are recorded before an explicit guarded apply. Cleanup uses
`git worktree remove` only after accept or discard, and stale entries can be
recovered with `git worktree list/prune`.

The in-process runner keeps the provider and OAuth token in Vera, gives each
agent a child JSONL session and cloned permission view, caps depth at one and
concurrency at four, and filters nested delegation. Write tasks receive one
generated worktree each; `subagent_result` returns bounded diff metadata and
only an explicit apply/discard decision can change or remove that worktree.

Source: [git-worktree documentation](https://git-scm.com/docs/git-worktree).

## Providers and images

The provider adapter keeps a provider-neutral input model. Text-only providers
must reject image parts with a capability error; image-capable adapters declare
support explicitly. Function calls remain client-executed and are returned as
function-call outputs on the next request. OpenAI Responses and xAI Responses
both document input arrays, function tools, and multimodal input as the
wire-level shape.

Sources: [OpenAI Responses quickstart](https://platform.openai.com/docs/quickstart/make-your-first-api-request),
[xAI function calling](https://docs.x.ai/developers/tools/function-calling),
and [xAI image context](https://docs.x.ai/developers/tools/advanced-usage).

## PTY

The process abstraction is deliberately isolated from the tool registry.
`portable-pty` 0.9 from the WezTerm ecosystem is the selected backend and is
used for the interactive master/slave session; there is no pipe fallback in the
release path. Process output is read into a bounded ring buffer, cursors are
monotonic, and environments are allow-listed. Unix PTY children establish a
session/process-group boundary so session close signals the whole development
server tree. The release gate verifies PTY cleanup on session close. No SQLite,
repository index, or persistent memory is introduced.

## Release fixtures and gates

Fixture tests cover malformed MCP schemas/results, notification routing,
timeouts/cancellation, pending-question resume, versioned plan revisions,
credential redaction (including repeated token fields), symlink escapes,
command-prefix normalization, bounded process output, PTY dimension validation,
CDP host denial, image limits, stale and oversized worktree diffs, and child
environment inspection. `scripts/benchmark-capabilities.sh` records release
binary size, warm startup, maximum resident set size when the host exposes it,
tool-schema token/byte cost, and the four-agent fixture wall time. Release runs
`cargo fmt --all`, clippy with warnings as errors, `cargo test`, the release
script, and live smoke tests where credentials are intentionally supplied only
by the parent process.
